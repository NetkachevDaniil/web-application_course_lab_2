# ЛР2: Работа с данными запроса

## 1) Что требовалось по заданию

В этой лабораторной нужно было сделать Flask-приложение, которое демонстрирует:

1. получение и отображение данных HTTP-запроса:
   - URL-параметры;
   - заголовки;
   - cookie;
   - параметры формы;
2. форму авторизации (логин/пароль), где после отправки введённые данные выводятся на страницу;
3. отдельную форму проверки телефона с POST-отправкой, где:
   - проверяется количество цифр (10 или 11 по правилу из задания);
   - проверяется допустимый набор символов;
   - в зависимости от ошибки выводится конкретный текст;
   - поле подсвечивается через `is-invalid`, а текст ошибки выводится через `invalid-feedback`;
   - при успехе номер приводится к виду `8-***-***-**-**`.

---

## 2) Как это реализовано в проекте

### Основные маршруты

- `/` — главная страница с навигацией.
- `/request-info` — вывод URL-параметров, заголовков и cookie.
- `/set-demo-cookie` — установка демо-cookie, чтобы можно было наглядно показать работу cookie.
- `/auth-form` — форма авторизации:
  - GET: показать форму;
  - POST: принять поля и вывести их пользователю.
- `/phone-check` — форма проверки номера:
  - GET: пустая форма;
  - POST: серверная валидация и либо ошибка, либо форматированный номер.

### Где берутся данные запроса

- URL-параметры: `request.args`
- Заголовки: `request.headers`
- Cookie: `request.cookies`
- Поля формы: `request.form`

### Логика валидации телефона

Проверка разбита на 2 этапа:

1. **Проверка допустимых символов**
   - Разрешены только цифры и символы: пробел, `(` `)` `-` `.` `+`.
   - Если есть посторонний символ, показываем:
     - `Недопустимый ввод. В номере телефона встречаются недопустимые символы.`

2. **Проверка количества цифр**
   - Сначала выделяются только цифры.
   - Если номер начинается с `+7` или `8`, ожидается 11 цифр.
   - Во всех остальных случаях — 10 цифр.
   - При нарушении показываем:
     - `Недопустимый ввод. Неверное количество цифр.`

Если ошибок нет, номер форматируется в вид:
- `8-123-456-78-90`

### Почему такой порядок проверок

Сначала символы, потом длина — чтобы сообщение об ошибке было точным:
- если пользователь ввёл буквы, он сразу получает ошибку про недопустимые символы;
- если символы допустимы, но цифр не хватает/слишком много — получает ошибку по длине.

## 3) Техническая часть (что применялось)

- `Flask` — обработка маршрутов и шаблонов.
- `Jinja2` — рендер HTML-шаблонов.
- `Bootstrap 5` — базовая стилизация формы и валидации.
- Серверная валидация в Python (регулярные выражения + обработка цифр).

### Про Bootstrap-валидацию

- `is-invalid` добавляется на `<input>` при ошибке.
- `invalid-feedback` выводит текст ошибки под полем.
- Это соответствует требованию задания и стандартной практике Bootstrap.

## 5) Пошаговый сценарий показа на защите

1. Открыть главную страницу и показать, что есть отдельные пункты меню под каждый функционал.

   - валидный: `+7 (123) 456-75-90` -> форматированный номер;
   - невалидный по символам: `123abc` -> ошибка про символы;
   - невалидный по длине: `12345` -> ошибка про количество цифр.

## 7) Вопросы, которые могут задать, и готовые ответы

### В: Почему не ограничиться HTML-валидацией?
О: HTML-валидацию легко обойти, поэтому обязательна проверка на сервере. Сервер — единственный источник истины.

### В: Разница между `request.args` и `request.form`?
О: `request.args` — query-параметры в URL (обычно GET), `request.form` — тело формы при POST.

### В: Что такое cookie и где она хранится?
О: Cookie хранится в браузере пользователя, отправляется на сервер с каждым запросом к соответствующему домену/пути.

### В: Почему форматируем номер на сервере?
О: Чтобы получить единообразный вид данных независимо от того, как пользователь их ввёл.

## 8) Типичные ошибки при запуске

- Команда `pip` не найдена:
  - использовать `py -m pip` или `.\.venv\Scripts\python.exe -m pip`.
- Ошибка пути к Python в PowerShell:
  - путь должен быть без лишних пробелов (`..\.venv\...`, а не `..\ .venv\...`).
